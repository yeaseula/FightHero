# 1 프로젝트 소개
부트캠프에서 JavaScript Class 문법과 상속(extend) 을 학습한 뒤,<br />
이를 실제로 활용해 보고 싶다는 생각에서 시작한 미니 프로젝트입니다.<br /><br />

강사의 “미니게임을 한 번 만들어보라”는 제안을 계기로,<br />
객체지향 개념을 게임 로직에 적용하는 것을 목표로 제작했습니다.<br />

# 4 게임 개요
유저는 용사 팀과 괴수 팀에서 각각 하나의 캐릭터를 선택한 뒤<br />
턴제 방식으로 결투를 진행합니다.<br /><br />

공격 버튼을 번갈아 누르며 전투가 진행되고,<br />
상대방의 HP를 먼저 0 이하로 만든 쪽이 승리합니다.<br />

# 5 캐릭터 설계

모든 캐릭터는 공통된 기본 스탯(hp, atk)을 가지며<br />
기본 캐릭터 클래스를 상속(extend) 하여<br />
각 캐릭터마다 고유한 특성을 부여했습니다.<br /><br />

예를 들어:<br />

> 탱커
> 높은 체력과 방어 특화
> 상대 공격력을 50% 감소시키는 패시브 능력 보유

>힐러
>공격력은 낮지만 3회 공격마다 랜덤 확률로 HP를 최대치까지 회복
<br />
이처럼 공통 로직은 부모 클래스,<br />
캐릭터별 개성은 자식 클래스에서 관리하도록 설계했습니다.

# 6 ux 연출 포인트

단순한 전투 로직에 그치지 않고, 게임의 몰입감을 높이기 위한 연출에도 신경 썼습니다.<br />

- 캐릭터 선택 화면 → 실제 판타지 게임처럼 웅장하고 진지한 분위기
- 전투 화면 → 동일한 캐릭터를 귀엽고 캐주얼한 스타일로 변환
<br />
예를 들어,<br />
선택 화면에서는 거대하고 위협적인 괴수였던 캐릭터가<br />
전투 장면에서는 귀여운 고양이 캐릭터로 등장하도록 구성했습니다.
<br /><br />
이를 통해 긴장감과 재미의 대비를 의도했습니다.

- DOM 조작 기반 턴제 게임 로직 구현

# 8 에셋 제작 및 AI 활용

- AI 활용

판타지 게임 콘셉트에 맞는 캐릭터 에셋을 구하기 어려워<br />
AI 이미지 생성 도구를 활용해 캐릭터 이미지를 직접 제작했습니다.<br /><br />

프롬프트를 통해 세계관에 맞는 분위기, 캐릭터의 역할(탱커, 힐러, 괴수 등)을 명확히 전달해 원하는 스타일의 이미지를 생성했고,<br />
이를 캐릭터 선택 화면과 전투 화면에 맞게 서로 다른 톤의 이미지로 활용해 게임의 몰입감을 높였습니다.<br />

실제 사용한 프롬프트 일부입니다. <br />

```
마법사 프로필 이미지가 필요해.
사이즈 3x4 비율의 jpg 확장자로 부탁해.
가로사이즈 최소 150px이상 (위 생성한 이미지와 동일한 사이즈면 더 좋음)
이미지 주제는 '마법사' 이고 판타지게임이나 영화에 나오는것같은 이미지가 필요.
성별은 상관없음.
신비롭고 비밀스러운 분위기의 마법사였으면 좋겠어. 마법사 하면 떠오르는 보라색or초록빛을 메인컬러로 잡아줘.
```
<br />

- 에셋 제작
웹 서비스에 바로 사용할 수 있도록 iPad Pro의 Procreate 앱을 활용해 배경 제거 및 간단한 후처리를 진행했습니다.<br />

전문 디자인 툴에 얽매이기보다,<br />
당시 가장 빠르게 결과를 낼 수 있는 도구를 선택해 서비스 완성도를 높이는 데 집중했습니다.

# 9 회고

Fight Hero는 JavaScript에서 Class 문법을 처음으로 실제 프로젝트에 적용해 본 경험이었습니다.<br />
그 전까지는 class가 단순한 문법처럼 느껴졌고,<br />
“굳이 왜 써야 하지?”라는 의문이 명확히 해소되지 않은 상태였습니다.<br /><br />

하지만 캐릭터 수가 점점 늘어나면서<br />
공통된 속성(hp, atk)과 동작을 하나의 틀로 묶는 방식이<br />
확실히 효율적이라는 것을 체감할 수 있었습니다.
<br /><br />
만약 캐릭터가 2~3개 수준이었다면<br />
각각을 개별적으로 설계하는 것도 가능했겠지만,<br />
8개의 캐릭터를 만들며 공통 로직을 반복 작성하는 것은<br />
현실적인 선택이 아니라는 점을 느꼈습니다.<br /><br />

이 과정에서 class는 단순한 문법이 아니라 <br />
확장을 고려한 설계 도구라는 인식을 갖게 되었습니다.
<br /><br />

다만 아쉬운 점도 분명히 남았습니다.<br />

처음에는 캐릭터 하나만 만들 생각으로 시작했기 때문에<br />
기본 클래스 내부에 기능을 비교적 많이 담아두었고,<br />
결과적으로 기본 클래스가 다소 무거워진 구조가 되었습니다.
<br /><br />
프로젝트를 진행하며 캐릭터 수가 늘어나자,<br />
“공통 기능”과 “개별 특성”을 더 명확히 분리했어야 했다는 아쉬움이 남았습니다.
<br /><br />
지금 다시 만든다면 기본 클래스는 최대한 가볍게 유지하고,<br />
캐릭터별 특성이나 조건부 로직은 보다 명확하게 분리하는 구조를 선택했을 것 같습니다.
<br /><br />
그럼에도 불구하고 이 프로젝트는
처음부터 거창한 규모가 아닌 미니 프로젝트로 시작했기 때문에,<br />
초심자의 입장에서 객체지향 개념을 부담 없이 실험해 볼 수 있었던 점에서 의미 있는 경험이었습니다.
<br />
Fight Hero를 통해 JavaScript 문법을 “아는 것”과 실제로 로직을 설계해 “사용해 보는 것”의 차이를 체감했고,<br />
이 경험은 이후 프로젝트에서 구조와 역할을 먼저 고민하는 사고방식으로 자연스럽게 이어지게 되었습니다.
